## Problem Statement

用户在手机端（微信、浏览器）发现有价值的内容（文章、视频、图片），通过"文件传输助手"转发给自己。但这些素材大量堆积，从未被整理、消化、利用。用户说：

> "我变成了知识的囤积者，并没有理解、消化、利用知识。"

现有工具"笔记同步助手"能解决同步问题，但数据经过商业公司服务器，用户不信任，且需要付费。用户的核心诉求：

> "数据绝对安全地从手机端传输到电脑端 Obsidian，不经过第三方服务器。功能宁可少一点。最主要最核心的就是要保证数据安全。"

## What Makes This Cool

这不是又一个"稍后读"工具。这是一个**零信任知识管道**：

1. **数据完全私有** — 从手机到电脑，内容不经过任何外部服务器。Syncthing P2P 加密直传，端到端 TLS 1.3。
2. **手机端零处理** — 只保存原始素材。AI 处理、全文抓取全部在 Obsidian 中完成。手机端是"捕获工具"，Obsidian 才是"思考工具"。
3. **开源透明** — MIT 许可。任何人都能审计代码，确认没有数据泄露。
4. **文件即协议** — 手机和电脑不直接通信。Markdown 文件的 YAML frontmatter 就是通信协议。这是最简单的跨设备通信方式——不需要 API、不需要端口、不需要鉴权。

这个概念的价值在于：**你分享的不是"链接"，你分享的是未来自己的一份知识资产。** 半年后你在 Obsidian 里搜索一个概念，能搜到那天在微信看到的那篇文章——全文可搜索、标签已打、摘要已写。这就是从"囤积"到"消化"的转变。

## Constraints

1. 数据不经过任何第三方服务器（自部署服务器可以接受）
2. 手机端入口灵活——微信转发、浏览器分享、任何 App 的分享菜单
3. 功能少而精——核心是安全传输，不做花里胡哨
4. 开源（MIT）
5. 只支持 Android + Windows Obsidian 桌面端（v1）

## Premises

来自 Phase 3，全部确认：

1. **P1: 用 Syncthing 做传输层，不自己造轮子。** — 已确认。Syncthing 解决了 NAT 穿透、加密、增量同步，自己写需要数千行代码且不如它稳定。

2. **P2: 手机端只保存原始素材，处理在 Obsidian 中完成。** — 已确认。这是关键设计决策。手机 App 是"快门"，Obsidian 是"暗房"。

3. **P3: AI 处理必须是可选的，核心价值是安全同步。** — 已确认。不配 API Key 就跳过 AI，同步功能必须独立可用。

## Approaches Considered

### Approach A: Syncthing P2P + Android App + Obsidian 插件 (SELECTED)
  Summary: Syncthing 做文件传输，Android App 接收分享存 Markdown，Obsidian 插件监听处理
  Effort:  M (human: ~5 days / CC: ~2 hours)
  Risk:    Low (所有组件都是成熟技术)
  Pros:
    - 零服务器，数据完全本地，这是最强的隐私保证
    - Syncthing 全球数百万用户验证过的稳定性
    - 三个组件完全解耦，各干各的，互不依赖
  Cons:
    - 用户需要额外安装 Syncthing（增加入门门槛）
    - 非局域网场景通过 Syncthing 社区中继服务器(relays)传输，延迟会增加，但依然可用
    - Android App 需要处理 Scoped Storage 权限
  Reuses: Syncthing（现成的 P2P 同步工具）

### Approach B: 自部署服务 + PWA
  Summary: 自己写一个轻量 HTTP 服务跑在电脑上，手机通过浏览器 PWA 发送内容
  Effort:  M (human: ~7 days / CC: ~3 hours)
  Risk:    Med (需要处理外网访问、认证、HTTPS)
  Pros:
    - 不需要额外安装 Syncthing
    - 可以有更好的 UI（网页表单）
    - 支持外网访问（不在家也能同步）
  Cons:
    - 需要维护一个服务进程
    - 外网访问需要配置端口转发/DDNS/HTTPS 证书
    - 服务器挂了就完全不可用

### Approach C: iLink 协议直连
  Summary: 逆向微信 iLink 协议，Obsidian 插件直接跟微信服务器通信
  Effort:  L (human: ~14 days / CC: ~5 hours)
  Risk:    High (微信协议非公开，随时可能失效)
  Pros:
    - 最酷的方案——不需要手机装任何额外 App
    - 双向同步（可以发消息回微信）
  Cons:
    - 微信协议变更就全挂
    - 逆向工程有法律风险
    - 技术门槛高，维护成本大

## Recommended Approach

**方案 A (Syncthing P2P)** — 因为它是唯一同时满足"零服务器"和"成熟可靠"两个硬约束的方案。方案 B 的内网穿透是运维噩梦，方案 C 的协议依赖是定时炸弹。

### 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| Android App | Kotlin + Jetpack Compose, minSDK 26 | 接收 Android Share Intent，透明主题秒开 |
| 传输层 | Syncthing v2.1+ | P2P TLS 1.3 加密，用于 LAN 直连 + 全球中继 |
| Obsidian 插件 | TypeScript + chokidar | 监听收件箱文件夹，处理管道，可选 AI |
| 协议 | Markdown YAML Frontmatter | 手机与电脑通过文件的 frontmatter 元数据"通信" |

### 系统架构

```
┌────────── 手机端 ──────────┐      ┌────────── 电脑端 ──────────┐
│                              │      │                              │
│  微信/浏览器 → [分享]        │      │  E:\obsidian\obsidian-Inbox\ │
│       ↓                      │      │       ↓                      │
│  Quick Capture App           │      │  InboxWatcher (chokidar)     │
│  ├ ContentClassifier  识别类型│      │       ↓                      │
│  ├ MarkdownGenerator  生成MD  │      │  Pipeline (处理管道)         │
│  └ FileWriter → Syncthing目录 │      │  ├ ContentClassifier 识别类型│
│       ↓                      │      │  ├ ContentFetcher 全文抓取(v2)│
│  Syncthing ──P2P TLS──→      │      │  ├ AIProcessor AI标签(v2)    │
│                              │      │  └ Mover 归档到正式目录       │
└──────────────────────────────┘      └──────────────────────────────┘
```

### 处理管道步骤

1. **检测**: chokidar 监听新增 .md 文件（awaitWriteFinish: 2s + 防抖 500ms）
2. **解析**: 提取 YAML frontmatter → 读取 `source_type`, `url`, `status` 字段
3. **分类**: `link` → 全文抓取(v2) / `image` → 移附件 / `video` → 字幕抓取(v2) / `file` → 移附件 / `plain` → 格式化
4. **AI 处理(v2, 可选)**: 若开启 AI → 生成标签(3-5个) + 摘要(2-3句话)
5. **移动**: 收件箱 → `输出目录/YYYY-MM-DD/文件名.md`

### Frontmatter Schema（手机→电脑协议）

```yaml
---
# 必填字段 (手机端生成)
title: "素材标题"           # 从分享内容提取，未命名则为"未命名素材"
source_type: "link"         # link | image | video | file | plain
source: "weixin.qq.com"     # 来源域名
created: "2026-06-29-143052" # 创建时间戳
status: "pending"           # pending → processing → processed → error

# 可选字段 (手机端按需填充)
url: "https://..."          # 如果有链接
attachment: "image.jpg"     # 如果有附件

# 电脑端处理后添加的字段
processed: "2026-06-29-143102"
tags: [AI, 跨境电商]        # AI 生成(v2)
summary: "本文讨论了..."    # AI 生成(v2)
ai_generated: true           # 标记是否经过AI
---
正文内容（原始素材：URL、文字、文件引用等）
```

### 异常场景处理

| 场景 | 处理策略 |
|------|---------|
| Syncthing 未运行 | 文件保存在手机本地目录，Syncthing 启动后自动同步 |
| 网络断开 | Syncthing 自动排队，网络恢复后同步 |
| 同步冲突 | Syncthing 自动生成 `.sync-conflict` 文件，插件跳过冲突文件 |
| 存储空间不足 | Android: FileWriter 返回 null，Toast 提示用户 |
| 插件处理失败 | 保留原文件在收件箱，status→error，不删除 |

### v1 核心 vs v2 扩展

| 版本 | 范围 |
|------|------|
| **v1 (核心)** | Syncthing 配对引导、Android 分享接收、Markdown 生成、Obsidian 插件监听+归档 |
| **v2 (扩展)** | 公众号全文抓取(高难度⚠️)、B站/YouTube字幕、AI标签摘要、iOS支持、F-Droid上架 |

## Open Questions

1. **iOS 支持**: 当前仅 Android。iOS 的分享机制不同（Share Extension），需要 Swift 重写。需求有多大？
2. **多设备**: 如果用户有多台电脑/多部手机，同步逻辑需要处理设备 ID 去重
3. **Syncthing 的替代**: 如果未来有更好的 P2P 同步工具出现，是否迁移？

## What I noticed about how you think

- 你说"功能宁可少一点，不需要做太多花里胡哨的功能"——这句话定义了整个项目的设计哲学。每个功能都是因为"没有它就不可用"才加的，不是因为"有它更酷"。

- 你对"笔记同步助手"的担忧不是功能不够，而是"数据会经过他们的服务器，不安全"——隐私不是锦上添花的需求，是你的硬约束。这让项目从"又一个同步工具"变成了"零信任知识管道"。

- 你在选择方案时选了已经实现好的方案 A，而不是更酷的方案 C——实用主义优先于炫技。这是好工程师的直觉。

- 你先要了 Syncthing 配对，再要了代码，现在要文档——你在按正确顺序做事：基础设施 → 实现 → 文档归档。不是"写完代码就跑"。
